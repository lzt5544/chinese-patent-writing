# 中文专利撰写 (Chinese Patent Writing)

## Overview

**当你被触发时，你的任务是：为用户生成一份格式规范的中文专利申请 Word (.docx) 文件。**

帮助用户撰写符合中国《专利法》《专利法实施细则》及《专利审查指南》（2023年修订版，AI部分2025年进一步细化）要求的中文专利申请文件。

**核心原则：** 先判断专利类型，再按法定结构撰写，确保充分公开、权利要求清楚有层次、以说明书为依据。

**底层逻辑链（贯穿全文）：** 背景技术缺陷（技术问题）→ 技术方案（区别特征）→ 有益效果（技术进步）。三者必须一一对应、环环相扣。

**工作流程（每次被触发后按此执行）：**
1. 理解用户技术方案 → 2. 按流程撰写专利内容 → 3. 将内容组织为 JSON → 4. **自我审查与迭代优化**（见下方完整章节）→ 5. **调用 `scripts/generate_patent_docx.py` 生成 `.docx`** → 6. 将文件交付给用户

> **永远不要输出纯文本或 Markdown 给用户。** 最终交付物必须是通过脚本生成的 `.docx` Word 文件。

> **不要跳过自我审查步骤。** 在生成 `.docx` 之前，必须对照审查清单逐项检查并修复问题。宁可多一轮迭代，不可将有明显缺陷的文件交付用户。

## 输出格式

本 skill **输出 `.docx` Word 文档**，格式符合中国国家知识产权局（CNIPA）《专利审查指南》要求：

| 格式项 | 设置 |
|--------|------|
| 纸张 | A4（210mm × 297mm），纵向 |
| 页边距 | 上25mm / 左25mm / 右15mm / 下15mm |
| 字体 | 宋体（中文）+ Times New Roman（西文） |
| 正文字号 | 小四号（12pt，字高约4.23mm） |
| 行距 | 固定值18pt |
| 颜色 | 黑色 |
| 页码 | 页脚居中 |
| 首行缩进 | 两个汉字（0.74cm） |

**文档结构**：一个 `.docx` 文件包含权利要求书、说明书、说明书摘要三部分，用分页符分隔。权利要求书顶格书写无缩进；说明书正文首行缩进两字符。**说明书不得含有段落编号（如 `[0001]`）**，权利要求用阿拉伯数字编号 `1. 2. 3.`。

**生成流程**：撰写专利内容 → 按 JSON 结构组织 → 调用 `scripts/generate_patent_docx.py` 生成 `.docx` → 交付用户。

> 依赖：`pip install python-docx`

## 范围说明

- **适用：** 中国发明专利、实用新型专利的完整撰写、交底书转化、单章撰写、审查优化
- **不适用：** 外国专利、外观设计专利、纯检索分析（不撰写）

## 专利类型判断

| 维度 | 发明专利 | 实用新型专利 |
|------|---------|-------------|
| 保护客体 | 产品、方法或其改进 | 产品的形状、构造或其结合 |
| 创造性要求 | 突出的实质性特点和显著进步 | 实质性特点和进步 |
| 保护期限 | 20年 | 10年 |
| 是否必须附图 | 必要时 | 必须有附图 |

**判断要点：** 涉及方法、工艺流程、软件算法、材料配方 → 发明专利；仅涉及产品结构改进 → 可考虑实用新型。

### 同日申请

根据专利法第9条，同一发明创造可**同日**申请发明和实用新型。产品结构改进同时涉及方法 → 强烈建议同日申请。同日申请的两份独立权利要求保护范围应完全相同。

### 术语规范

说明书全文根据专利类型统一使用术语：发明专利 → "本发明"；实用新型 → "本实用新型"。**不得混用**。

## 完整撰写流程

> 撰写前先完成[专利类型判断](#专利类型判断)。涉及方法/算法/材料配方 → 发明；仅产品结构 → 可考虑实用新型。

```
1. 理解技术方案 → 2. 构建权利要求 → 3. 确定名称
→ 4. 撰写说明书 → 5. 撰写摘要 → 6. 自我审查与迭代优化
```

### 1. 理解技术方案

从用户描述或技术交底书提取核心三要素：
- **要解决的技术问题**：现有技术存在什么缺陷？（背景技术中的缺陷 = 技术问题）
- **技术方案**：具体如何解决？关键结构/步骤是什么？**注意：发明人提供的交底书通常是具体的实施例，需要从中提炼并概括上位为权利要求的保护范围**
- **有益效果**：相比现有技术有什么优势？（量化优先，且必须与背景技术的缺陷一一对应）

交底书信息提取模板：
```
【发明名称】（如有）：________________
【技术领域】：________________
【现有技术】：现有方案及缺陷（每条缺陷对应一个技术问题）
【本发明方案】：整体概述、关键结构/步骤/模块、与现有技术的区别、替代方案
【有益效果】：量化效果1/2/3
【附图信息】：几幅图，分别展示什么
【实施例】：最佳实施方式、关键参数/数值范围、材料/工艺条件
```

### 2. 构建权利要求

**撰写顺序：** 先根据背景技术的缺陷确定技术问题，再围绕技术问题构建独立权利要求的技术方案，最后推导有益效果。确保问题→方案→效果三者一一对应。

**核心原则：每项权利要求必须是一个完整的技术方案。权利要求最重要的是**清楚**——权利要求界定专利权的保护范围，边界不清则无法判定侵权。

**独立权利要求**（必须）：前序部分 + 特征部分（"其特征在于..."），包含全部必要技术特征
**从属权利要求**（建议2-5项）：引用在前权利要求 + 附加技术特征

**术语概括技巧（权利要求上位化）：**

| 概括类型 | 含义 | 示例 |
|---------|------|------|
| **选择型概括** | 上位词代替多个下位词，保护任一个下位概念 | 卡接、螺纹连接、锁扣连接 → **可拆卸连接** |
| **合并型概括** | 母集代替所有子集，保护由子集组成的整体 | 公卡扣、母卡扣 → **卡扣组件** |
| **并列概括** | 用"或"/"和/或"将多个特征写入同一权利要求 | "还包括D组分**或**E组分" / "还包括D组分**和/或**E组分" |

**主题名称选择策略：**
- 确定主题名称时，充分考虑可能的侵权者身份（制造商、销售商、使用者），尽量将所有环节的侵权主体纳入保护范围
- 必要时针对不同侵权者构建不同主题名称的权利要求（如方法+装置+系统）
- **大产品包含小产品的改进时**：先将**具有改进且可单独销售的最小产品**作为第一主题（如油封），再将包含它的更大产品作为第二主题（如电机）。这样可对上下游全面维权，同时防止大厂商逃避侵权责任

关键规则：
- 默认**开放式**连接词 "包括"（化学组合物用"由...组成"）
- **禁用模糊用语**：约、左右、基本上、大致、较好、最好是、高温
- 避免纯功能性限定，除非无法用结构定义且说明书有支持
- **方法类权利要求必须用分步格式 S1/S2/S3...**，每步一个关键技术动作
- 一项高质量的权利要求，应仅通过文字描述就能再现还原与附图基本接近的结构或流程

### 3. 确定发明名称

一般不超过25字（≤40字），禁用：人名/地名/商标/型号、"及其类似物""及其他"等含糊词。
✅"一种自适应调光护眼台灯及其控制方法" ❌"一种新型装置及其相关方法"

### 4. 撰写说明书

**⚠️ 核心规则：正文用连续段落，禁用项目符号列表和序号词。** 多个效果用分段叙述或"进一步地"衔接，不用"第一，...；第二，..."罗列。

**⚠️ 分步阐述 + 分段规则：**
- **技术方案**中每个 S 步骤**必须独占一个自然段**（前后各空一行），紧扣独立权利要求展开。可包含**必要的核心公式**（如函数关系、判断条件），但不需要枚举完整参数定义和取值范围——那是实施例的事。
- **实施例**中每个 S 步骤**必须独占一个或多个独立段落**，S 标题在 Word 中加粗显示，含完整数学公式和**全部**参数定义。

**⚠️ 技术方案 vs 实施例的区别：**

| | 技术方案（发明内容中） | 实施例（具体实施方式中） |
|------|------|------|
| 作用 | 对权利要求的技术方案展开说明 | **证明**技术方案确实能解决所述技术问题并产生预期效果；让技术人员能照着做出来 |
| 格式 | S1/S2/S3 独占段落 | **S1：** **S2：** 加粗标题，独占段落 |
| 公式 | ✅ 核心关系式、判断条件、约束条件 | ✅ 所有公式 + **完整参数定义和取值范围** |
| 参数定义 | ❌ 不需要给定具体值 | ✅ 每个变量都定义，含单位、范围 |
| 范例 | S1：采集邮件文本数据，对所述邮件文本进行分词处理并过滤停用词，得到有效词项集合。 | **S1：** 设原始邮件文本为 D，分词器使用结巴分词算法，得到词序列 T=(t₁,…,tₖ)。过滤停用词集合 S_stop={s₁,…,s_p} 后，保留有效词项 W={wᵢ\|wᵢ∈T, wᵢ∉S_stop}，\|W\|=n。 |
| 范例 | S2：基于所述有效词项集合，采用 TF-IDF 算法提取特征向量，其中分量 fᵢ = tf(tᵢ)·log(N/df(tᵢ))。 | **S2：** 对 W 计算 TF-IDF 向量 v=(v₁,…,vₘ)，其中 vᵢ = tf(tᵢ,D)·log(N/df(tᵢ))，tf(tᵢ,D) 为词项 tᵢ 在文档 D 中的词频，N=100000 为语料库文档总数，df(tᵢ) 为包含该词项的文档数，通过预计算词典查询。 |

**⚠️ 实施例数学语言规则：** 具体实施方式中所有计算逻辑、判断条件、参数关系必须用完整数学公式表达。禁止用自然语言模糊描述（如"根据环境光强弱自动调节亮度"）。包括函数关系式、条件判断式、取值范围、迭代公式等。

按五章节撰写：
1. **技术领域**（1-2句）
2. **背景技术**（1-3段，200-400字）：简述最接近现有技术→客观指出缺陷。**不写行业/市场/发展史**

   **撰写深度策略：** 根据发明难易程度和新创性调整背景技术的详细程度：
   - **技术方案复杂、新创性高** → 详细写明现有技术的缺陷，帮助审查员理解发明的技术贡献
   - **技术方案简单、新创性低** → 将现有技术的问题适当上位概括，避免给出技术启示
   - **背景技术中的缺陷 = 本申请要解决的技术问题**，是整篇专利的逻辑起点
3. **发明内容**：
   - **技术问题（1-2段）**：**不写"本发明所要解决的技术问题是提供一种…"**，这不是技术问题而是技术方案的开头。正确写法是从背景技术的缺陷自然引出，聚焦核心技术矛盾。
     - ✅ 好的写法：`现有台灯无法根据环境光照变化自动联动调节亮度与色温，导致暗环境下高色温光线易引发视疲劳，且用户手动调节操作繁琐。`
     - ❌ 差的写法：`本发明所要解决的技术问题是提供一种自适应调光护眼台灯，以解决现有台灯无法自动调节的技术问题。`（空泛、套路化、与技术方案混淆）
     - **原则：背景技术里写了什么缺陷，技术问题就直接点明要解决什么，一一对应，绝不绕弯子。**
   - **技术方案（≥S 步骤数个独立段落）**：每个 S 独占一段，段落间空行分隔
   - **有益效果（1-3段）**：**必须推导，不能断言。** 每个效果都要有"因为采用了技术特征X，所以产生了效果Y"的推导逻辑链，且与背景技术的缺陷一一对应。
     - ✅ 好的写法：`护眼曲线算法实现了亮度与色温的联动调节，环境光变暗时自动提高亮度并切换暖色调，减少了人眼瞳孔的频繁调节，降低了视疲劳程度。`
     - ❌ 差的写法：`有效减少视疲劳，使用便捷。`（纯断言、无推导、混入用户体验）
     - **禁止混入商业/用户体验效果**：不说"市场前景广阔""使用便捷""成本低""用户体验好"，只写技术层面的客观效果
     - **推导链模板**：`本发明通过[区别技术特征]，实现了[技术效果]，因为[物理/化学/电学原理]`。效果放在前面，原理跟在后面，避免长句导致读者找不到落点。
     - **量化优先**：能用数据不用形容词。不说"显著提高"，说"效率提高30%""寿命延长至1500小时"
     - **不用"第一/第二/第三"罗列**：说明书正文禁用序号词。多个效果用**分段**或**"进一步地"**衔接。参见 `references/effects-patterns.md` 中的五种写法类型。
4. **附图说明**（每图一句）
5. **具体实施方式**：每个实施例独立成块，实施例内用 **S1：**、**S2：**……分步骤展开，每步独占段落。方法类每步含完整数学公式和参数定义。产品类给出具体结构参数。覆盖全部权利要求特征。**实施例的核心功能不仅是"教别人怎么做"，更是证明该技术方案确实能解决所述技术问题并产生预期效果。** 因此，实施例不仅要给出操作步骤和参数，还应包含实施结果或效果验证（如实验数据、性能测试、对比表格等），以证明技术方案的有效性。

   **实施例的两种常用写法：**

   **写法一（单图多方案对比）：** 在同一幅图中展示发明人提供的方案 A' 和代理师挖掘的替代方案 A''。先写 A' 的具体结构和原理，再写 A'' 的具体结构和原理，最后对比两者的优劣。

   **写法二（多图各一方案）：** 实施例一用图X展示方案 A'（含完整部件功能描述和协同原理），实施例二用图Y展示方案 A''（同样展开）。当权利要求保护范围较宽时，至少给出两个不同实施例；涉及数值范围时，还应给出中间值实施例。

### 5. 撰写摘要

≤300字，包含名称、领域、方案要点、主要用途。附图标记加括号。

## 单独撰写专利某一部分

当用户只要求某一部分时，**不要输出完整文件**，只输出该部分。

| 专利部分 | 可独立？ | 需用户提供 |
|---------|---------|-----------|
| **权利要求书** | ✅ | 完整技术方案、要解决的技术问题、现有技术状况 |
| **技术领域** | ✅ | 发明名称、所属领域 |
| **背景技术** | ✅ | 现有技术状况及缺陷 |
| **发明内容** | ⚠️ | 技术方案、有益效果（建议有权利要求草稿） |
| **附图说明** | ✅ | 附图清单 |
| **具体实施方式** | ⚠️ | 详细实现方案（建议参考权利要求） |
| **摘要** | ✅ | 发明名称、方案要点、主要用途 |

**修改已有内容时：** 先确认修改意图（扩大范围？补全公开？修正错误？），再按审查标准逐项检查。修改必须遵守专利法第33条——不得超出原说明书和权利要求书记载的范围。

### 单独撰写输出格式

当用户只要求某一部分时，直接输出该部分的 **Word 文档**（调用脚本生成单独的 `.docx`），不生成完整专利文件。

JSON 结构中只包含用户要求的部分。例如单独撰写权利要求书：

```json
{
  "patent_type": "发明专利",
  "invention_name": "发明名称",
  "sections": {
    "claims": [
      "一种[产品/方法名称]，其特征在于，包括...",
      "根据权利要求1所述的[产品/方法名称]，其特征在于..."
    ]
  }
}
```

然后调用脚本：
```bash
python scripts/generate_patent_docx.py claims_only.json output_claims.docx
```

> **提示**：本部分为独立章节。后续撰写其他部分时，保持术语一致，确保权利要求与说明书相互支持。

## 完整输出流程

### 步骤1：撰写专利内容

按上述撰写流程（理解技术方案 → 构建权利要求 → 确定名称 → 撰写说明书 → 撰写摘要）完成内容创作。

### 步骤2：组织为 JSON 结构

将撰写好的内容按以下 JSON 结构组织。这是调用脚本生成 Word 的输入格式：

```json
{
  "patent_type": "发明专利",
  "invention_name": "发明名称",
  "sections": {
    "claims": [
      "一种[产品/方法名称]，其特征在于，包括/包含/具有...",
      "根据权利要求1所述的[产品/方法名称]，其特征在于..."
    ],
    "specification": {
      "tech_field": "本发明涉及XXX技术领域，具体涉及一种XXX。",
      "background": "现有技术中最接近的方案及其缺陷...",
      "invention_content": {
        "problem": "现有台灯无法根据环境光照变化自动联动调节亮度与色温，在暗环境下高色温光线容易导致视疲劳，且用户手动调节操作繁琐、精度低。",
        "solution": "技术方案正文（方法类用 S1/S2/S3 分步阐述，可包含必要核心公式，不展开完整参数定义）...",
        "effects": "本发明通过[区别特征]实现[效果1]，[推导逻辑]。与现有[对比对象]相比，[指标]由[旧值]提升至[新值]，[效果2]。进一步地，[效果3]。[结论句，证明方案确实有效]。"
      },
      "figure_desc": "图1是XXX的结构示意图；图2是XXX方法的流程图。",
      "embodiment": "具体实施方式正文（实施例1：... 实施例2：...）"
    },
    "abstract": {
      "text": "摘要正文（≤300字）...",
      "figure": "图1"
    }
  }
}
```

**注意：**
- `claims` 数组的每个元素是一项权利要求的完整文本（不含编号，脚本自动添加 `1. 2. 3.`）
- 说明书正文**不使用 Markdown 格式符号**（如 `**加粗**`、`- 列表`），直接写纯文本，脚本负责 Word 排版
- 技术方案中的 S 步骤用自然段分隔，实施例中的 S 步骤用空行分隔
- **公式用 `$...$` 包裹 LaTeX 语法**，脚本自动转换为 Word 原生公式。支持：下标 `_`、上标 `^`、分数 `\frac{a}{b}`、根号 `\sqrt{a}`、希腊字母 `\delta` `\alpha` 等、运算符 `\cdot` `\times` `\pm` `\leq` `\geq` `\neq` `\infty` 等
- 示例：`$L_{target}(\delta) = L_{max} \cdot (1-\delta/\delta_{max}) + L_{min} \cdot (\delta/\delta_{max})$`

### 步骤3：自我审查与迭代优化

将 JSON 保存到临时文件，运行自动验证：

```bash
python scripts/validate_patent_json.py patent_content.json --output review_report.json

根据审查报告逐项修复问题。修复后重新验证，最多迭代 **3 轮**。验证通过（0 errors）后进入下一步。

### 步骤4：生成 Word 文档

```

脚本会自动设置：A4纸张、专利标准页边距、宋体小四号、固定行距18pt、页脚居中页码、首行缩进两字符。

## 专利法基本要求

1. **充分公开**（第26条第3款）：说明书以所属技术领域的技术人员能够实现为准
2. **权利要求清楚/支持**（第26条第4款）：权利要求书应以说明书为依据，清楚、简要地限定保护范围
3. **单一性**（第31条）：一件申请限于一项发明；属于一个总的发明构思的例外
4. **实用性**（第22条第4款）：能够制造或使用，并能产生积极效果
5. **新颖性**（第22条第2款）：不属于现有技术

## 专利修改的法律限制（专利法第33条）

> **对专利申请文件的修改不得超出原说明书和权利要求书记载的范围。**

✅ 可以：从原始记载中直接导出的、对原始内容缩限/澄清/优化表达的
❌ 不可以：新增未记载的技术特征、下位扩大为上位、改变技术方案本质

## 自我审查与迭代优化

> **此步骤是生成 .docx 之前的必经环节，不得跳过。**

撰写完成后，必须对专利内容进行全面自我审查。发现问题后迭代修复，直至通过审查或达到最大轮次。

### 审查流程

```
撰写内容 → 组织 JSON → 自动验证 → 人工审查 → 修复问题 → 重新验证 → 生成 .docx
                                  ↑__________________________________|
                                          迭代循环（最多3轮）
```

### 执行步骤

#### 第1步：保存 JSON 并运行自动验证

将撰写好的专利内容保存为临时 JSON 文件，运行验证脚本：

```bash
python scripts/validate_patent_json.py patent_content.json --output review_report.json
```

验证脚本自动检查 6 大类共 40+ 条规则，输出结构化审查报告。

#### 第2步：逐项分析审查报告

阅读审查报告，将问题分类处理：

| 问题类型 | 处理方式 |
|---------|---------|
| **error** | 必须修复。包括：模糊用语、序号词、商业用语、术语不一致、格式缺陷、章节缺失 |
| **warning** | 建议修复。包括：背景技术过长/过短、缺少推导逻辑、效果未量化、实施例缺少验证 |

对每个 error/warning：
1. 定位到具体章节和行
2. 根据 `suggestion` 字段提供的修复建议进行修改
3. 如 suggestion 不够具体，参考下方审查维度中的详细规范

#### 第3步：修复问题

按优先级修复：
1. **致命缺陷**（章节缺失、权利要求为空）→ 补全内容
2. **术语规范**（本发明/本实用新型混用、模糊词、序号词）→ 逐一替换
3. **格式规范**（缺少"其特征在于"、项目符号列表）→ 调整格式
4. **质量提升**（效果无推导、实施例无验证、背景过长）→ 重写相关内容

> **修复原则**：每个问题都要实际修改 JSON 中的文本内容，不要只标注而不修改。

#### 第4步：重新验证

修复完成后再次运行验证脚本：

```bash
python scripts/validate_patent_json.py patent_content.json --output review_report.json
```

如果仍有 error，重复第2-3步，最多迭代 **3 轮**。

#### 第5步：通过后进入人工审查

自动验证通过（0 errors）后，还需进行一轮 **人工审查**（即你作为撰写代理的语义层面审查）。自动验证只能检查硬性规则，以下维度需要人工判断：

**逻辑链审查（最重要）：**
- [ ] 背景技术的每个缺陷 → 对应一个技术问题
- [ ] 每个技术问题 → 对应独立权利要求中的必要技术特征
- [ ] 每个区别技术特征 → 对应有益效果中的推导
- [ ] 有益效果的每个效果点 → 在背景技术中有对应的缺陷
- [ ] 用箭头法验证：缺陷₁ → 问题₁ → 特征₁ → 效果₁ 是否一一贯通

**充分公开审查：**
- [ ] 实施例是否能让技术人员无需创造性劳动就能实现？
- [ ] 关键参数是否全部定义？（单位、范围、取值依据）
- [ ] 计算逻辑是否用数学公式而非自然语言描述？
- [ ] AI/算法类发明是否披露了模型结构、训练步骤、参数？

**保护范围审查：**
- [ ] 独立权利要求是否包含了全部必要技术特征？（不多不少）
- [ ] 独立权利要求是否没有包含非必要技术特征？（不应缩小保护范围）
- [ ] 从属权利要求是否形成了层次化的保护梯度？
- [ ] 主题名称是否覆盖了所有侵权环节？（制造者、销售者、使用者）

**写作质量审查：**
- [ ] 有益效果是否用推导而非断言？（"通过X→实现Y→因为Z"）
- [ ] 技术问题是否聚焦核心技术矛盾？（非套路化的"提供一种…"）
- [ ] 背景技术是否简练（1-3段、直击缺陷）？（无行业/市场背景）
- [ ] 是否有长句过度使用"由于"开头导致读者等待？

### 迭代退出条件

| 条件 | 处理 |
|------|------|
| 0 errors + 0 warnings | ✅ 质量优秀，进入人工审查 |
| 0 errors + N warnings | ✅ 通过，人工审查时关注 warning 项 |
| 少量 errors（1-5），第3轮后 | ⚠️ 记录剩余问题，在交付时告知用户 |
| 大量 errors（>5），第3轮后 | ⚠️ 可能存在系统性问题，检查是否对用户技术方案理解有误 |

### 审查维度速查表

自动验证覆盖的 6 大维度：

| 维度 | 检查项数 | 典型问题 |
|------|---------|---------|
| 格式完整性 | 8 | 章节缺失、权利要求为空、摘要超长 |
| 术语规范 | 5 | "约"/"左右"/"本发明"vs"本实用新型" |
| 禁用模式 | 6 | 序号词、商业用语、套路化问题表述、"由于"开头 |
| 逻辑一致性 | 4 | 问题↔缺陷↔效果对应、S步骤一致性 |
| 结构规范 | 7 | 权利要求格式、背景技术长度、项目符号 |
| 实施例质量 | 5 | 分步格式、效果验证、数学语言、参数定义 |

> 完整规则详见：`references/checklist.md`、`references/writing-specs.md`、`references/effects-patterns.md`

## 与用户协作要点

### 完整撰写
1. **先确认技术理解**：用自己的话复述技术方案，确认理解正确
2. **询问附图**：实用新型必须有附图，发明专利视情况
3. **明确保护重点**：了解用户最看重保护的技术点
4. **分阶段交付**：先交付背景技术+技术问题+权利要求书框架（确保问题→方案对应） → 用户确认 → 再交付完整说明书。背景技术是整篇专利的逻辑起点，优先确认背景技术和技术问题有助于后续权利要求的准确构建

### 单独撰写某一部分
1. **明确范围**：只输出用户要的部分
2. **索要上下文**：询问是否有其他部分草稿，确保术语一致
3. **修改意图确认**：扩大范围？补全公开？修正错误？
4. **提醒协调**：修改可能波及其他部分
