# 常见错误与自检清单

## 常见错误

| 错误 | 后果 | 修正方法 |
|------|------|---------|
| 权利要求缺少必要技术特征 | 无法解决技术问题，被驳回 | 确保独立权利要求包含全部必要特征 |
| 说明书公开不充分 | 被驳回或无效 | 至少一个实施例足以让技术人员实现 |
| 权利要求得不到说明书支持 | 被驳回 | 权利要求的概括应在说明书中有实施例支撑 |
| 使用模糊用语（约、左右、基本上） | 权利要求不清楚，被要求修改 | 用精确数值或明确描述替代 |
| 使用商业宣传用语 | 被要求修改 | 用客观技术语言替代："高效"→"效率提高30%" |
| 权利要求范围过窄 | 保护价值低 | 上位概括非必要特征，下位用于从属权利要求 |
| 多个独立发明合案 | 缺乏单一性，被要求分案 | 一件申请只限一个总的发明构思 |
| 从属权利要求引用在后权利要求 | 形式缺陷，需补正 | 只能引用编号更小的（在前的）权利要求 |
| 多项从属权利要求引用另一多项从属权利要求 | 形式缺陷，需补正 | 多项从属不得作为另一多项从属的基础 |
| 附图标记不一致 | 补正或驳回 | 同一组成部分在全文使用相同标记 |
| 数值范围缺乏实施例支撑 | 权利要求得不到支持，被驳回 | 说明书中至少给出范围端点和优选值的实施例 |
| 软件专利写成纯代码权利要求 | 不属于专利保护客体，被驳回 | 写成方法步骤或功能模块装置的权利要求 |
| 修改超出原始记载范围 | 修改不被接受，维持原缺陷 | 严格依据原说明书和权利要求书记载内容修改 |
| 使用项目符号/自动编号列表撰写说明书正文 | 格式不规范，不属于专利正式文体 | 全部改为连续段落叙述；效果罗列用分段或"进一步地"衔接，禁用序号词 |
| 背景技术过于冗长啰嗦 | 审查员跳过不看，关键缺陷被淹没 | 控制在1-3段、200-400字，只写最接近的现有技术及其缺陷 |
| 背景技术写行业宏观分析/市场前景 | 被视为商业宣传或无关内容 | 直接描述现有技术方案及其缺陷，不写行业发展/市场背景 |
| 编造专利号、公开号、论文题名或申请人 | 背景技术失真，影响新颖性/创造性判断，严重降低文件可信度 | 未核验时只作概括描述；具体文献需经检索确认后再引用 |

## 完整撰写后的自检清单

完成撰写后，按以下清单逐项检查。

### 权利要求书自检

- [ ] 独立权利要求包含了解决技术问题的**全部必要技术特征**
- [ ] 独立权利要求中没有包含**非必要技术特征**
- [ ] 使用了**开放式**连接词（"包括"），除非化学组合物需用封闭式
- [ ] 没有出现模糊用语（约、左右、基本上、大致、较好、最好是、高温）
- [ ] 没有出现商业宣传用语
- [ ] 从属权利要求只能引用**在前的**权利要求
- [ ] 多项从属权利要求以**择一方式**引用，且**不作为**另一多项从属的基础
- [ ] 科技术语与说明书中使用的**一致**
- [ ] 每项权利要求结尾仅用**句号**
- [ ] 有从属权利要求形成**层次化保护梯度**

### 说明书自检

- [ ] **技术领域**具体、准确，不过宽不过窄（1-2句话即可）
- [ ] **背景技术**控制在**1-3段**，简练直击缺陷，无行业宏观分析/市场背景
- [ ] **背景技术**客观指出了现有技术缺陷，且缺陷**仅限于**本发明要解决的
- [ ] **背景技术**没有编造专利号、公开号、论文题名、申请人、发布日期或具体检索结论
- [ ] **发明内容**中有"要解决的技术问题""技术方案""有益效果"三个子节
- [ ] **有益效果**与背景技术中的缺陷**一一对应**
- [ ] **有益效果**尽量量化（百分比、数值、倍数）
- [ ] **具体实施方式**至少有一个**详细实施例**
- [ ] **具体实施方式**覆盖了权利要求书中的**全部技术特征**
- [ ] 说明书正文**全部使用连续段落叙述**，没有出现无序列表（`-` 或 `*` 项目符号）
- [ ] 没有使用"如权利要求……所述的……"等引用语
- [ ] 没有使用商业性宣传用语
- [ ] 附图标记在全文**一致**

### 摘要自检

- [ ] 不超过**300字**（含标点）
- [ ] 包含发明名称、技术领域、技术方案要点、主要用途
- [ ] 没有包含权利要求中**没有**的内容
- [ ] 没有商业宣传用语

### 整体一致性检查

- [ ] 发明名称在全文各处**一致**
- [ ] 权利要求书中的技术特征在说明书中有**对应描述**
- [ ] 说明书中的技术方案不**超出**权利要求的保护范围
- [ ] 权利要求不**超出**说明书公开的范围（支持关系）

## 语义审查自检清单（多子代理审查层）

以下清单与 4 个专家子代理的审查维度一一对应，供主代理在子代理不可用时作为 fallback 使用，也可用于交叉验证子代理输出。

### LogicChainReviewer — 逻辑链闭环

- [ ] 背景技术中提到的**每个具体缺陷**都在技术问题中有明确对应
- [ ] 每个技术问题都在独立权利要求的**必要技术特征**中得到了解决
- [ ] 独立权利要求中的**每个区别技术特征**都在有益效果中有对应推导
- [ ] 有益效果中的**每个效果点**都能回溯到背景技术中的某个缺陷
- [ ] 用箭头法验证：缺陷₁ → 问题₁ → 特征₁ → 效果₁ 是否一一贯通

### EnablementReviewer — 充分公开

- [ ] 具体实施方式足以让所属领域技术人员**无需创造性劳动**即可实现
- [ ] 所有关键参数都给出了**定义、单位、取值范围或取值依据**
- [ ] 计算逻辑、判断条件、约束关系用**完整数学公式**表达，而非自然语言模糊描述
- [ ] 具体实施方式**覆盖了权利要求书中的全部技术特征**
- [ ] 实施例包含效果验证（实验条件、测试数据、对比结果）
- [ ] AI/软件类：披露了模型结构、训练数据、训练步骤、关键超参数
- [ ] 化学/材料类：披露了原料配比、反应条件、测试方法
- [ ] 机械/电子类：披露了结构参数、连接关系、控制逻辑

### ScopeReviewer — 保护范围

- [ ] 独立权利要求包含了解决技术问题的**全部必要技术特征**（不多不少）
- [ ] 独立权利要求没有包含**非必要技术特征**（避免不当缩小保护范围）
- [ ] 从属权利要求形成了**点 → 线 → 面**的层次化保护梯度
- [ ] 主题名称覆盖了可能的侵权环节（制造者、销售者、使用者）
- [ ] 涉及方法+装置/系统的发明，布局了不同主题名称的权利要求
- [ ] 权利要求的概括得到了说明书实施例的**支持**

### WritingQualityReviewer — 写作质量

- [ ] 有益效果采用推导式表达："通过X → 实现Y → 因为Z"
- [ ] 技术问题聚焦核心技术矛盾，不使用"所要解决的技术问题是提供一种…"等套路化表述
- [ ] 背景技术简练（1-3段、200-400字），直击缺陷，无行业宏观分析/市场背景
- [ ] 避免句子以"由于"开头导致读者长时间等待主句
- [ ] 全文科技术语统一，无"的的不休"、口语化表达
- [ ] 摘要包含名称、领域、方案要点、主要用途四要素
- [ ] 说明书正文使用连续段落叙述，无项目符号/自动编号列表

### 冲突处理

- [ ] 不同子代理对同一位置给出相反建议时，已在 `conflicts` 中标记
- [ ] 主代理已对冲突项进行综合判断，选择最符合专利法和支持关系的方案
