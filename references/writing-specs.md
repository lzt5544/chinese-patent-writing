# 说明书与权利要求书撰写规范

> 以下为《专利审查指南》详细要求。五部分法定结构及长度/格式规则见主 SKILL.md，此文件补充细节。

## 技术方案与实施例的分步阐述规则

方法/流程/算法类发明的技术方案和具体实施方式，**必须用 S1、S2、S3... 逐步骤阐述**。

### 铁律：每个 S 独占段落

| 规则 | 正确 | 错误 |
|------|------|------|
| S 标题加粗 | S 步骤标题加粗显示 | S 步骤标题无加粗，与正文混排 |
| 每个 S 独占段 | S1 和 S2 之间**必须空一行** | S1 和 S2 连在一起，仅用句号或分号分隔 |
| 段落内部不分点 | S1 段落内可嵌套 `(1) ... (2) ...` 子点 | 整个技术方案一个大段，S1/S2/S3 混在一起 |

### 技术方案中的分步格式

技术方案是权利要求的展开说明，**用 S1/S2/S3 分步，可包含必要的核心公式，但不需要完整参数定义和取值范围**（留给实施例）。

每个 S 步骤说清楚三件事：①做了什么、②核心公式/关系（如有）、③输出流向哪里。

**❌ 错误（公式满天飞，每个变量都定义，写成实施例了）：**
> **S1：** 采集邮件文本数据 D，进行分词处理得到 W = {w₁, w₂, …, wₙ}，
> 其中 n = |W| 为有效分词数量。过滤停用词 S_stop = {s₁, s₂, …, s_p}，
> 保留 W' = {wᵢ | wᵢ ∈ D, wᵢ ∉ S_stop}，|W'| = m。
>
> **S2：** 对 W' 计算 TF-IDF 向量 v = (f₁, f₂, …, fₘ)，其中分量
> fᵢ = tf(tᵢ, D) · log(N / df(tᵢ))，tf(tᵢ, D) = count(tᵢ, D) / Σ count(tⱼ, D)，
> N = 100000，df(tᵢ) 通过预计算词典查询。

**✅ 正确（必要的核心公式保留，但不过度展开参数）：**
>
> S1：采集邮件文本数据，对所述邮件文本进行分词处理并过滤停用词，
> 得到有效词项集合。
>
> S2：基于所述有效词项集合，采用 TF-IDF 算法提取特征向量，其中
> 特征向量的每个分量 fᵢ = tf(tᵢ) · log(N / df(tᵢ))，tf(tᵢ) 为对应
> 词项在邮件文本中的词频，N 为语料库文档总数，df(tᵢ) 为包含该词项
> 的文档数。
>
> S3：将所述特征向量输入预先训练的 SVM 分类模型，根据分类结果
> 判断所述邮件是否为垃圾邮件。

**技术方案公式的"度"：**
- ✅ 核心函数关系：`L_target(δ) = L_max · (1 - δ/δ_max) + L_min · (δ/δ_max)` — 这就是护眼曲线的核心思想
- ✅ 判断条件：当环境光 δ 降低时，L_target 单调递增且色温向暖色调偏移
- ❌ 具体数值：`L_max = 500 lux`、`δ_max = 1000 lux`、`τ_C = 100K` — 这些留给实施例

### 实施例的分步分点规则

实施例**必须**以分步分点形式展开，条理清晰：

1. 实施例以"实施例N：[简述]"开头，独立成段
2. 内部用"S1：""S2：""S3："逐一展开（Word中自动加粗），**每个 S 独占一个或多个段落**
3. S 步骤内部可嵌套 `(1) ... (2) ...` 子点来展开子步骤或参数说明
4. 多个实施例之间用空行分隔

**实施例结构模板：**
```
**实施例1：**

**S1：** [第一段——步骤概述与核心公式]

[第二段（可选）——参数定义与取值范围]

**S2：** [第一段——步骤概述与核心公式]

(1) 子步骤A：[描述]，满足 [公式]。

(2) 子步骤B：[描述]，满足 [公式]。

**S3：** [内容...]

**效果验证（方法/产品类均适用）：**
[给出实验条件、测试数据、对比结果，证明本实施例的技术方案确实能解决
所述技术问题并产生预期效果。可包含：实验环境设置、测试样本量、性能指标、
与现有技术的对比表格、统计学分析等]

**实施例2（可选）：**

[变更XXX参数/结构后的另一种实施方式，同样包含S步骤展开和效果验证]
```

### 实施例的效果验证要求

实施例的核心功能不仅是"教别人怎么做"，**更是证明该技术方案确实能解决所述技术问题并产生预期效果**。因此，每个实施例在给出操作步骤和参数之后，应当包含效果验证部分。

**效果验证的撰写要点：**

| 验证要素 | 方法类发明 | 产品类发明 |
|---------|-----------|-----------|
| **实验/测试条件** | 输入数据集规模、硬件环境、软件版本 | 测试环境温度、湿度、负载条件 |
| **实施过程** | 按S步骤执行后的输出结果 | 按产品结构组装后的工作状态 |
| **性能指标** | 准确率、效率、误差、收敛速度等量化数据 | 强度、寿命、功耗、精度等量化数据 |
| **对比验证** | 与现有技术方案在同一条件下的对比结果 | 与现有产品在同一测试条件下的对比结果 |
| **结论** | 数据表明本方案确实解决了所述技术问题 | 数据表明本产品确实产生了预期效果 |

**效果验证示例（台灯）：**
```
在照度为50 lux的暗环境和800 lux的亮环境下分别测试本实施例台灯
与现有固定亮度台灯各2小时。测试结果表明：暗环境下，本实施例自动将亮度
提升至400 lux、色温降至3000K，受试者视疲劳评分由6.8分降至3.2分；
亮环境下，本实施例自动将亮度降至250 lux、色温升至5500K，眩光不适感
评分由5.5分降至2.1分。上述结果证明，本发明通过护眼曲线算法实现的
亮度与色温联动调节，确实能够在不同环境光照条件下有效降低视疲劳和眩光风险。
```

**❌ 错误（实施例不分步，所有内容搅在一起）：**
> 本实施例中，首先设置采样周期为100ms，光线传感器采集环境光照度 δ，
> MCU 读取 δ 后根据公式 L_target = L_max·(1-δ/δ_max) + L_min·(δ/δ_max)
> 计算目标亮度，然后转换为 PWM 占空比输出给驱动电路……

**✅ 正确（实施例逐步骤展开，每步独占段落）：**
>
> **实施例1：**
>
> **S1：** 设置采样周期 T_s = 100ms。光线传感器通过 ADC 采集环境光照强度 δ，
> δ ∈ [0, δ_max]，δ_max = 1000 lux。为避免瞬时波动引起频繁调节，定义触发条件：
> 连续 k = 3 个采样周期均满足 |δ_t - δ_{t-1}| / δ_{t-1} > ε 时才进入 S2，
> 其中敏感度系数 ε = 0.10。
>
> **S2：** MCU 控制器读取当前 δ，根据护眼曲线计算目标亮度 L_target(δ) 和
> 目标色温 C_target(δ)。计算公式如下：
>
> (1) 目标亮度：L_target(δ) = L_max · (1 - δ/δ_max) + L_min · (δ/δ_max)，
> 其中 L_max = 500 lux，L_min = 100 lux，分别对应最大和最小亮度。
>
> (2) 目标色温：C_target(δ) = C_warm + (C_cool - C_warm) · (δ/δ_max)，
> 其中 C_warm = 2700K，C_cool = 6500K。
>
> (3) 调节死区判断：若 |L_target - L_current| < τ_L 且 |C_target - C_current| < τ_C，
> 则跳过本次调节。其中 τ_L = 5 lux，τ_C = 100K。
>
> **S3：** MCU 控制器将 (L_target, C_target) 转换为两路 PWM 占空比信号
> (D₁, D₂)，其中 D₁ = L_target / L_max，D₂ = (C_target - C_warm) / (C_cool - C_warm)。
> LED 驱动电路根据 (D₁, D₂) 分别控制冷白光组和暖白光组的驱动电流
> I₁ = I_ref · D₁，I₂ = I_ref · D₂，实现亮度和色温的联合调节。

## 实施例的数学语言要求

具体实施方式中所有计算逻辑**必须用数学公式表达，不得用自然语言模糊描述**。

| 禁止写法 | 正确写法 |
|---------|---------|
| "根据环境光强弱自动调节亮度" | 定义亮度调节函数 L(δ) = L₀ · (1 - δ/δ_max)，其中 L₀ 为基准亮度，δ 为当前环境光照度，δ_max 为最大环境光照度 |
| "当差值较大时触发调节" | 若 |T_target - T_current| > τ，则执行色温调节；其中 τ 为触发阈值，τ = 500K |
| "采用加权平均的方式融合" | y = Σ_{i=1}^{n} wᵢ · xᵢ，其中 wᵢ = exp(-dᵢ) / Σ_{j=1}^{n} exp(-dⱼ) |
| "经过多次迭代后收敛" | 迭代至 ||w_{t+1} - w_t||₂ < 10⁻⁶ 时停止，其中 w_{t+1} = w_t - η_t · ∇L(w_t)，η_t = η₀ / (1 + γ·t) |
| "在一定范围内随机选择" | ξ ∼ U(ξ_min, ξ_max)，其中 U 为均匀分布，ξ_min = 0.1，ξ_max = 0.9 |

**参数定义必须完整：** 公式中出现的每个变量都要给出定义和取值范围，不得出现未定义符号。

## 背景技术简练规则

背景技术是审查员理解发明起点和动机的关键部分，**切忌冗长啰嗦**：

**长度与格式：** 1-3段、200-400字，连续段落，禁用无序列表。

**内容要求：** 简述最接近的现有技术 → 客观指出具体缺陷（每个缺陷对应一个技术问题）→ 尽可能说明缺陷的技术原因。

**禁止：** 行业宏观分析/市场背景/技术发展史、列举不相关现有技术、本发明的优点、商业宣传用语。

**背景技术写法对比：**

```
❌ 啰嗦写法（应避免）：
随着我国经济社会的快速发展和人民生活水平的不断提高，消费者对农产品质量安全的
关注度越来越高。传统农产品供应链存在信息不透明、数据易篡改、追溯效率低等问题，
严重制约了消费者对农产品的信任。近年来，国家出台了多项政策鼓励农产品信息化建设。
在此背景下，区块链技术作为一种分布式账本技术，具有去中心化、不可篡改、可追溯
等特点，为解决农产品溯源问题提供了新的技术路径。然而，现有的基于区块链的溯源
方案仍存在上链数据真实性问题……

✅ 简练写法（推荐）：
现有技术中，公开号为CNXXXXXX的中国专利公开了一种基于区块链的农产品溯源系统，
该系统在各流通环节采集数据后直接上传至区块链。该方案存在以下缺陷：各环节数据
由各自作业人员人工录入，缺乏统一的数据采集标准，导致上链数据的真实性和一致性
难以保证，消费者无法确认链上数据与实际产品是否对应。
```

**发明内容格式规则：**
- 正文必须以**连续段落**形式叙述，每一子节（要解决的技术问题、技术方案、有益效果）分别用1-4段自然段描述
- **不得使用**项目符号列表（Word 中不用自动编号/项目符号功能）
- 如确需罗列多个效果点，用**分段**叙述或"进一步地"衔接，禁用"第一/第二/第三"等序号词

### 充分公开标准（专利法第26条第3款）

> 说明书应当对发明或者实用新型作出清楚、完整的说明，以所属技术领域的技术人员能够实现为准。

**"能够实现"的含义：** 所属技术领域的技术人员按照说明书记载的内容，无需创造性劳动，就能够实现该发明或实用新型的技术方案，解决其技术问题，并产生预期的技术效果。

**实务要点：**
- 不需要对发明的每个细节均一一列明，未记载的技术细节若属于公知常识或常规手段，不会导致公开不充分
- 但核心技术逻辑、关键参数、实现路径必须清楚披露
- AI/算法类发明需记载模型模块、层级、连接关系及训练步骤、参数

### 引证文件规范

| 要求 | 具体规定 |
|------|---------|
| **公开性** | 引证文件须为公开出版物（含电子出版物） |
| **非专利文件/外国专利** | 公开日须**在本申请申请日之前** |
| **中国专利文件** | 公开日**不能晚于本申请的公开日** |
| **外国文件** | 应以原文文字写明出处，必要时给出中文译文（括号内） |
| **禁止用语** | 不得使用"如权利要求……所述的……"等引用语；不得使用商业性宣传用语 |

### 用语与格式

- **用词规范、语句清楚**；禁用商业宣传用语（"市场前景良好""首创""革命性"）；不对原因/理由作不必要说明
- **正文用连续段落**：技术领域、背景技术、发明内容、具体实施方式必须以自然段落行文
- **禁用项目符号/自动编号列表**（Word 中不用项目符号或自动编号功能），附图说明罗列图号除外
- **效果罗列用分段或"进一步地"衔接**：禁用"第一/第二/第三"等序号词，多个效果分段叙述
- **每段 3-6 句**，避免超过 200 字的"砖墙式"文本
- 涉及核苷酸/氨基酸序列的，提交符合 WIPO 标准的序列表（单独部分，单独编页码）

## 权利要求书详细撰写规范

### 核心法律依据（专利法第26条第4款）

> 权利要求书应当以说明书为依据，清楚、简要地限定要求专利保护的范围。

三个核心要件：**清楚**、**简要**、**以说明书为依据（支持）**

### 一、清楚

"清楚"体现在三个层面：

| 层面 | 要求 | 示例 |
|------|------|------|
| **类型清楚** | 产品权利要求用结构/组成特征；方法权利要求用工艺/步骤特征 | 产品是"包括...的装置"，方法是"包括以下步骤..." |
| **保护范围清楚** | 使用明确的技术术语，避免含义不确定的用语 | ✅ "48℃-52℃" ❌ "大约50度左右" |
| **引用关系清楚** | 从属权利要求引用关系明确、逻辑清晰 | 从属权利要求只能引用在前的权利要求 |

**模糊用语黑名单：** 约、左右、基本上、大致、较好、最好是、高温、适当、必要时、优选

### 二、简要

1. **表述简要**：除技术特征外不作不必要描述，不使用商业性宣传用语
2. **整体简要**：权利要求数目合理，避免两项以上保护范围实质相同的同类权利要求

### 三、以说明书为依据（支持）

- 权利要求概括的技术方案应是技术人员能从说明书中**直接得到或合理概括得出**的
- 权利要求的保护范围**不得超出**说明书公开的范围
- 说明书应记载足够详细的实施例，以支持权利要求中的上位概括

### 四、必要技术特征（专利法实施细则第23条）

> **必要技术特征**：为解决技术问题所**必不可少**的技术特征，其总和足以构成技术方案，使之区别于背景技术的其他技术方案。

**独立权利要求结构（前序部分 + 特征部分）：**

```
前序部分：主题名称 + 与最接近现有技术共有的必要技术特征
特征部分："其特征在于..." + 区别于最接近现有技术的必要技术特征
```

**撰写原则：**
- 必须记载解决技术问题的**全部**必要技术特征，不得遗漏
- 不应包含非必要技术特征（如"杯身颜色为蓝色"），以免不当缩小保护范围
- 在涵盖所有实施方式的前提下，对技术特征进行合理上位概括，谋求最大保护范围

### 五、从属权利要求（专利法实施细则第25条）

```
引用部分："根据权利要求X所述的[主题名称]，"
限定部分："其特征在于，[附加技术特征]"
```

**规则：**
- 只能引用**在前的**权利要求
- 引用两项以上的多项从属权利要求，只能以**择一方式**引用（"根据权利要求1或2所述..."），不得作为另一项多项从属权利要求的基础
- 编号用阿拉伯数字顺序编号

### 六、形式要求

| 项目 | 规定 |
|------|------|
| **标点** | 每项权利要求结尾仅用句号 |
| **附图标记** | 可引用说明书附图标记，置于括号内；**不得解释为对权利要求的限制** |
| **公式/化学式** | 可以有化学式或数学式 |
| **插图/表格** | **不得有插图**，除绝对必要外不得使用"如说明书……部分所述"或"如图……所示" |
| **科技术语** | 应与说明书中使用的术语一致 |

## 附图要求

### 附图必备性

- **实用新型**：说明书**必须有附图**
- **发明专利**：必要时应当有附图

### 附图规范（专利法实施细则第21条）

- 用制图工具和**黑色墨水**绘制，线条均匀清晰，不得涂改
- 不得使用工程蓝图
- 按 **"图1，图2，……"** 顺序编号排列
- 说明书文字部分未提及的附图标记**不得**在附图中出现，反之亦然
- 同一组成部分的附图标记在申请文件中应当**一致**
- 附图中除必需的词语外，**不应当含有其他注释**（流程图、框图除外）
- 附图大小及清晰度应保证在缩小到 **4厘米×6厘米** 时，仍能清晰分辨图中各个细节

## 摘要详细要求

### 内容（专利法实施细则第26条）

- 写明发明或实用新型名称和所属技术领域
- 清楚反映所要解决的**技术问题**、技术方案的**要点**及**主要用途**
- 可以包含最能说明发明的化学式

### 形式

| 项目 | 要求 |
|------|------|
| **字数** | 文字部分（含标点符号）**不得超过300个字** |
| **附图标记** | 应加括号 |
| **商业用语** | 不得使用商业性宣传用语 |
| **法律效力** | 仅属技术情报，**不具有法律效力**，不能作为修改或解释权利要求的依据 |
| **摘要附图** | 有附图的申请，应在请求书中指定一幅最能说明技术特征的附图作为摘要附图 |

## 数值范围撰写规范

权利要求书和说明书中涉及数值范围时，遵循以下规范：

| 情形 | 写法 | 示例 |
|------|------|------|
| **闭区间** | "X至Y" | "温度为50℃至80℃" |
| **开区间** | "大于X且小于Y" | "温度大于50℃且小于80℃" |
| **半开区间** | "X以上" / "Y以下" | "温度为50℃以上" / "80℃以下" |
| **优选范围** | 写明端点值和优选值 | "温度为50℃至80℃，优选为60℃至70℃" |

**审查要点：**
- 说明书中应给出端点值附近的**实施例**以支持权利要求中的范围
- 避免使用"左右""约""大概"等模糊表述
- **禁止使用**"X-Y"或"X~Y"格式表示数值范围，必须使用"X至Y"格式
- 涉及多个参数的范围时，各参数的范围应**独立限定**，避免交叉限定导致保护范围不清
